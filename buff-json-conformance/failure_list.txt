# buff-json proto3 JSON conformance — known failures
#
# Each line is the full name of a conformance test that buff-json is expected to
# fail (or a `*` wildcard pattern). The conformance_test_runner treats listed
# tests as "expected to fail": a listed test that fails is OK, and a listed test
# that unexpectedly *passes* is reported as an error (so prune entries as they
# get fixed).
#
# Test name format (proto3 JSON only):
#   Required.Proto3.JsonInput.<case>.<OutputFormat>
#   Recommended.Proto3.JsonInput.<case>.<OutputFormat>
#   Required.Proto3.ProtobufInput.<case>.JsonOutput
#
# Curation status: encode-side gaps and the silent-data-corruption decode gaps
# (int32/uint32/string strictness, wrong repeated element types, top-level null,
# out-of-range Timestamp/Duration JSON input, empty Any @type) are FIXED, so they
# are not listed. The entries below are inputs buff-json intentionally does NOT
# reject — the decoded value is recognizable/correct and rejecting would cost more
# than the gap is worth. With these listed, the suite has no UNEXPECTED failures,
# so ENFORCE_CONFORMANCE can be flipped to 1 to gate CI on conformance.

# --- Tier 2: lenient acceptance — recognizable value decoded, rejection skipped ---
# Float/double JSON numbers that overflow the type are accepted as +/-Infinity
# rather than rejected. The overflow->Infinity mapping round-trips, and rejecting
# would add an isInfinite() guard to every float/double parse for a non-canonical
# input — treated as normal lenient behaviour.
Required.Proto3.JsonInput.FloatFieldTooLarge
Required.Proto3.JsonInput.FloatFieldTooSmall
Required.Proto3.JsonInput.DoubleFieldTooLarge
Required.Proto3.JsonInput.DoubleFieldTooSmall
# Duplicate members of the same oneof (e.g. {"oneofUint32":1,"oneofString":"test"})
# are accepted last-wins rather than rejected. Detecting duplicates requires
# per-oneof "already seen" state threaded through both decode paths and the
# codegen DecoderGenerator — too invasive for a single non-canonical case.
Required.Proto3.JsonInput.OneofFieldDuplicate

# --- Tier 3: non-canonical representation, value decoded is correct ---
# Integer accepted with a leading zero / plus sign (e.g. "01", "+1", "-01") — the
# parsed value is right (1, 1, -1); rejecting requires inspecting the raw number
# token, which fastjson2's lenient lexer has already normalized away.
Required.Proto3.JsonInput.Int32FieldLeadingZero
Required.Proto3.JsonInput.Int32FieldNegativeWithLeadingZero
Required.Proto3.JsonInput.Int32FieldPlusSign
# RFC 3339 timestamp accepted with a lowercase 't'/'z' separator — the parsed
# instant is correct; the spec mandates uppercase 'T'/'Z'.
Required.Proto3.JsonInput.TimestampJsonInputLowercaseT
Required.Proto3.JsonInput.TimestampJsonInputLowercaseZ
